# 사용자 데이터 이중 구조 — Firebase Auth + Firestore UserProfile

## Situation

Firebase Authentication은 로그인한 사용자를 `User` 객체로 관리한다.
이 객체에는 `uid`, `email`, `displayName` 같은 인증 정보만 포함되어 있다.
커머스 서비스에서는 닉네임, 배송지 주소, 포인트 등 앱 고유의 추가 정보가 필요했다.

또한 이메일/비밀번호 로그인과 Google 소셜 로그인을 모두 지원하는 상황에서
두 방식의 사용자 프로필 초기화 방식이 달라 분기 처리가 필요했다.

## Task

- Firebase Auth와 앱 데이터(Firestore)를 어떻게 연결할지 설계
- 소셜 로그인 사용자의 첫 로그인 시 프로필이 없는 경우를 처리해야 했다
- 인증 상태와 프로필 데이터를 앱 전체에 일관되게 공급하는 구조가 필요했다

## Action

사용자 데이터를 두 레이어로 분리했다.

```
Firebase Auth  →  User       (uid, email 등 인증 정보 — Firebase 관리)
Firestore      →  UserProfile (닉네임, 주소 등 앱 데이터 — 직접 관리)
```

`onAuthStateChanged`에서 인증 상태가 바뀔 때마다 두 데이터를 함께 로드하도록 `AuthProvider`를 설계했다.

소셜 로그인 분기는 `providerData`로 판별했다.

- `password` provider 존재 → 이메일 가입자 → Firestore에 이미 프로필 있음 → 조회만
- `password` provider 없음 → 소셜 가입자 → 첫 로그인 시 프로필 없을 수 있음 → 없으면 자동 생성

```ts
const isSocialOnly = !firebaseUser.providerData.some(
  (p) => p.providerId === "password"
);
```

`user`와 `profile` 두 값을 Context로 공급해 어느 컴포넌트에서든 `useAuth()`로 접근할 수 있게 했다.

## Result

**Auth와 UserProfile을 이중 구조로 분리하고 AuthProvider에서 통합 관리하는 방식을 채택했다.**

Firebase Auth는 인증·보안을 담당하고 Firestore는 앱 도메인 데이터를 담당하는
명확한 역할 분리가 가능해졌다. 소셜 로그인 사용자도 별도 처리 없이
첫 로그인 시점에 자동으로 프로필이 생성되어 일관된 사용자 경험을 제공한다.
`reloadProfile()`을 Context에 노출해 프로필 수정 후 즉시 갱신도 가능하다.
