# 인증 세션 유지 전략

## Situation

Firebase Authentication은 기본적으로 로그아웃 전까지 세션을 영구 유지한다.
JWT 기반의 Access Token(1시간 만료) + Refresh Token 구조를 SDK 내부에서 자동으로 처리하기 때문에
사용자는 토큰 갱신을 체감하지 못한다.

커머스 서비스를 구현하면서 "일정 기간이 지나면 재로그인을 요구하는 방식"을 도입할지 고민했다.
금융·의료 서비스에서 흔히 보이는 패턴으로, 보안을 강화할 수 있다는 장점이 있다.

## Task

- 커스텀 세션 만료(N일 후 강제 로그아웃)를 구현할 것인지 결정
- 사용자 경험과 보안 수준 사이의 균형점을 찾아야 했다

## Action

두 가지 선택지를 비교했다.

| | 기본 유지 (현재) | 커스텀 만료 |
|---|---|---|
| 구현 복잡도 | 낮음 | 로그인 시각 저장 + 체크 로직 필요 |
| UX | 편리 (자동 로그인) | 재로그인 마찰 발생 |
| 보안 필요 수준 | 커머스 수준으로 충분 | 금융·의료 수준 |
| 레퍼런스 | 무신사·쿠팡·네이버쇼핑 동일 방식 | — |

커스텀 세션 만료를 구현하려면 `localStorage`에 로그인 시각을 저장하고
`onAuthStateChanged` 콜백에서 경과 시간을 체크해 `signOut()`을 호출하는 방식이 필요하다.
Firebase Admin SDK 없이 클라이언트 단에서만 처리하면 클라이언트 조작에 취약하다는 한계도 있다.

## Result

**기본 세션 유지 방식을 채택했다.**

무신사·쿠팡 등 레퍼런스 서비스가 동일한 방식을 사용하고 있으며,
커머스 도메인에서 세션 만료로 인한 UX 저하는 이탈률 증가로 직결된다.
보안 강화가 필요하다면 Firebase 콘솔에서 Refresh Token 무효화(강제 로그아웃)를 사용하는 것이 더 안전한 접근이다.
