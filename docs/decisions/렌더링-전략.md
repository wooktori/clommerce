# 렌더링 전략 — SSR vs CSR 경계 설정

## Situation

Next.js App Router는 기본적으로 서버 컴포넌트(SSR)를 사용한다.
그러나 Firebase 클라이언트 SDK는 인증 토큰을 브라우저의 IndexedDB에 저장하기 때문에
서버는 요청 시점에 사용자가 로그인한 사람인지 알 수 없다.

HTTP는 무상태(Stateless) 프로토콜이라 서버는 매 요청마다 "이 사람이 누구인가"를
새로 확인해야 한다. 일반적인 인증 방식은 쿠키에 토큰을 담아 요청마다 서버로 전달하지만,
Firebase 클라이언트 SDK는 쿠키가 아닌 IndexedDB에 토큰을 저장하므로
서버 요청에 인증 정보가 포함되지 않는다.

## Task

- 어떤 페이지·기능을 SSR로 처리하고 어떤 것을 CSR로 처리할지 경계를 명확히 설정
- Firebase 클라이언트 SDK 구조에서 SSR의 한계를 파악하고 적절한 전략을 결정

## Action

로그인 상태 필요 여부를 기준으로 렌더링 전략을 나눴다.

| 영역 | 렌더링 | 이유 |
|---|---|---|
| 상품 목록 | SSR 가능 | 로그인 없이 접근 가능, SEO 필요 |
| 상품 상세 | SSR 가능 | 로그인 없이 접근 가능, SEO 필요 |
| 메인 페이지 | SSR 가능 | 공개 페이지 |
| 장바구니 | CSR | 로그인 상태 필요, SEO 불필요 |
| 구매 내역 | CSR | 로그인 상태 필요, SEO 불필요 |
| 판매자 상품 관리 | CSR | 로그인 상태 필요, SEO 불필요 |
| 헤더 유저 메뉴 | CSR | `onAuthStateChanged` 응답 후 확인 가능 |

현재 로그인 여부 확인 흐름:

```
사용자 요청
  → Next.js 서버: 인증 정보 없음, HTML 전송
  → 브라우저: JS 실행 → onAuthStateChanged → IndexedDB에서 토큰 확인
  → 로그인 상태 확인 후 데이터 fetch
```

Firebase Admin SDK + 쿠키 방식으로 전환하면 서버에서도 토큰 검증이 가능해져
로그인 상태가 필요한 영역도 SSR 처리가 가능하다. 그러나 구현 복잡도가 크게 올라간다.

## Result

**로그인 여부에 따라 SSR/CSR 경계를 나누는 전략을 채택했다.**

커머스에서 SEO가 중요한 영역(상품 목록·상세)은 SSR로 처리할 수 있어
Firebase 클라이언트 SDK의 제약이 치명적이지 않다.
장바구니·마이페이지·판매자 기능은 SEO가 필요 없는 영역이므로 CSR로 처리해도 무방하다.

Admin SDK 도입은 SSR 범위를 넓힐 수 있지만 구현 복잡도 대비 이점이 크지 않아
현재 프로젝트 범위에서는 클라이언트 SDK 유지가 적절한 판단이다.
