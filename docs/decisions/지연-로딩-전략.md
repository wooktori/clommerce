# 지연 로딩 전략: React.lazy vs next/dynamic

## Situation

Next.js App Router는 라우트 단위로 JS 번들을 자동 분리한다.
그러나 라우트 안에서 정적으로 import된 클라이언트 컴포넌트는 모두 동일한 청크에 포함된다.

현재 프로젝트에서 다음 의존성이 초기 번들에 포함되고 있었다:

- **react-hook-form + zod + @hookform/resolvers** (ProductForm, LoginForm, SignupForm)
- **react-intersection-observer** (SellerProductList)
- **Firebase auth 모듈** (LoginForm, SignupForm)

이 모듈들은 해당 컴포넌트가 렌더링되지 않는 상황(예: 로그인된 사용자가 판매자 상품 목록을 보는 경우)에도 번들에 포함되어 초기 로드 시간을 늘린다.

---

## Task

컴포넌트 단위로 JS 번들을 분리해 실제로 필요한 시점에만 로드되도록 지연 로딩을 적용한다.
동시에 로딩 중 스켈레톤 UI를 표시해 사용자 경험을 유지한다.

---

## Action: React.lazy vs next/dynamic 비교

### React.lazy + Suspense (순수 React 방식)

```tsx
import { lazy, Suspense } from "react";

const ProductForm = lazy(() => import("@/components/product/ProductForm"));

function Page() {
  return (
    <Suspense fallback={<Skeleton />}>
      <ProductForm />
    </Suspense>
  );
}
```

| 항목 | 내용 |
|---|---|
| 동작 위치 | 클라이언트 컴포넌트 내부에서만 사용 가능 |
| SSR | 지원 안 함 (SSR 중에 lazy 컴포넌트를 만나면 에러 또는 빈 렌더링) |
| fallback | Suspense로 직접 감싸야 함 |
| Next.js 통합 | 없음 (Next.js 최적화 파이프라인 밖) |

### next/dynamic (Next.js 공식 방식)

```tsx
import dynamic from "next/dynamic";

const ProductForm = dynamic(
  () => import("@/components/product/ProductForm"),
  { loading: () => <Skeleton /> }
);
```

| 항목 | 내용 |
|---|---|
| 동작 위치 | 서버 컴포넌트(page.tsx)에서도 사용 가능 |
| SSR | `ssr: false` 옵션으로 제어 가능 |
| fallback | `loading` 옵션으로 내장 처리 (Suspense 자동 추가) |
| Next.js 통합 | Next.js 번들러 최적화와 통합되어 청크 분리 보장 |

### 핵심 차이

**React.lazy는 클라이언트 런타임에서 동적 import를 처리**하므로 서버 컴포넌트에서 호출하면 의도대로 동작하지 않는다.

**next/dynamic은 Next.js 빌드 타임에 번들 분리를 처리**하므로 서버 컴포넌트(page.tsx)에서 선언해도 클라이언트 번들이 올바르게 분리된다. App Router에서 page.tsx는 서버 컴포넌트이므로 dynamic import는 반드시 next/dynamic을 써야 한다.

## Result

### 적용 결과

| 페이지 | 분리된 컴포넌트 | 포함 의존성 |
|---|---|---|
| `/seller/products` | SellerProductList | react-intersection-observer |
| `/seller/products/new` | NewProductClient | react-hook-form, zod |
| `/seller/products/[id]/edit` | EditProductClient | react-hook-form, zod |

로그인/회원가입 페이지는 적용 제외. AuthProvider가 이미 Firebase auth 모듈을 로드하므로 분리 효과가 없고, 폼 크기도 작아 스켈레톤으로 인한 CLS 리스크가 더 크다.

### 기대 효과

- 각 페이지의 서버 렌더링 HTML은 스켈레톤 UI가 포함된 채로 즉시 전달됨
- 컴포넌트 JS는 별도 청크로 분리되어 필요한 시점에 병렬로 로드됨
- 판매자 페이지를 방문하지 않는 구매자는 react-hook-form, zod 번들을 로드하지 않음

### 트레이드오프

- 컴포넌트 로드 전 짧은 스켈레톤 표시 구간 발생
- 스켈레톤 UI 형태가 실제 컴포넌트와 다를 경우 레이아웃 시프트(CLS) 가능성
- 작은 컴포넌트에 적용 시 네트워크 요청 증가로 오히려 불리할 수 있음 → 충분히 무거운 컴포넌트에만 적용
