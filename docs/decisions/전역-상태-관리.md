# 전역 상태 관리 전략 — Context vs Zustand vs URL

## Situation

커머스 앱을 개발하면서 관리해야 할 전역 상태가 여러 종류로 늘어났다.
초기에는 테마 하나만 고려했으나, 장바구니·모달·필터 등 성격이 다른 상태들이
추가될수록 단일 방식으로 모두 처리하기 어렵다는 것을 파악했다.

현재 인증 상태(`AuthProvider`)는 React Context로 관리 중이며,
Zustand 도입 여부와 상태별 적절한 관리 방식을 결정해야 했다.

## Task

- 장바구니·모달·필터 등 앞으로 늘어날 상태들을 어떤 방식으로 관리할지 결정
- 상태 종류별로 적합한 도구가 다를 수 있어 일괄 적용이 아닌 기준 정립이 필요했다

## Action

핵심 판단 기준을 하나로 정했다.

> **"이 상태가 바뀔 때 Provider 하위 전체가 리렌더링돼도 괜찮은가?"**

Context는 변경 시 Provider 하위 트리 전체가 리렌더링된다.
인증처럼 변경이 드문 경우엔 문제없지만, 장바구니처럼 자주 바뀌는 상태에 적용하면 성능 이슈가 생긴다.
Zustand는 해당 상태를 구독한 컴포넌트만 리렌더링되므로 UI 클라이언트 상태에 적합하다.
필터·검색어는 URL에 두면 공유·북마크·뒤로가기가 자연스럽게 동작한다.

상태별 비교:

| 상태 | 방식 | 이유 |
|---|---|---|
| 인증 | React Context | 변경 드물고 앱 전체가 동일하게 구독 |
| 테마 | React Context | 전체 트리 리렌더링이 의도된 동작 |
| 장바구니 | Zustand | 여러 컴포넌트가 독립적으로 읽고 쓰고, 변경 잦음 |
| 모달 | Zustand | 어디서든 열고 닫아야 하는 전역 UI 상태 |
| 필터 / 검색 | URL (useSearchParams) | 공유·북마크 가능, 뒤로가기 동작 자연스러움 |

## Result

**상태 종류에 따라 세 가지 방식을 병행하기로 결정했다.**

- **React Context**: 인증·테마처럼 변경이 드물고 앱 전체가 동일하게 참조하는 상태
- **Zustand**: 장바구니·모달처럼 변경이 잦고 여러 컴포넌트가 독립적으로 접근하는 UI 상태
- **URL**: 필터·검색어처럼 공유 가능하고 히스토리에 남아야 하는 상태

이 기준을 따르면 "모든 상태를 Context로" 또는 "모든 상태를 Zustand로" 처리하는
과설계나 과소설계 없이 상태 성격에 맞는 도구를 선택할 수 있다.
Zustand는 장바구니 기능 구현 시점에 도입한다.
