# 메인 페이지 렌더링 개선 — CSR → Server Prefetch + HydrationBoundary

## Situation

메인 페이지는 `렌더링-전략.md`에서 "SSR 가능" 영역으로 분류했고,
`메인-페이지-구현.md`에서도 SSR 적용을 목표로 설계했다.

그러나 UI 리뉴얼 과정에서 메인 페이지 구조가 카테고리 섹션 방식에서
"전체 상품 무한 스크롤 + 캐러셀 히어로" 방식으로 교체됐다.
이 과정에서 두 핵심 컴포넌트가 모두 클라이언트 컴포넌트로 구현됐다.

```
page.tsx (RSC) ← 서버 컴포넌트지만 아무 데이터도 fetch하지 않음
├── MainHero    ("use client") — useShopProducts(null) 호출
└── ShopPage    ("use client") — useShopProducts(queryCategory) 호출
```

두 컴포넌트가 모두 `useShopProducts`를 클라이언트에서 호출하므로
Next.js App Router를 쓰면서도 서버 렌더링을 전혀 활용하지 못하는 상태다.

### 구체적인 문제

**① 상품 목록이 초기 HTML에 없음**

브라우저가 JS를 파싱·실행하기 전까지 상품은 화면에 나타나지 않는다.
사용자는 반드시 skeleton을 경험하게 된다.
검색 엔진 크롤러도 상품 데이터를 읽지 못한다.

**② 이중 클라이언트 fetch**

`MainHero`와 `ShopPage`가 동일한 queryKey `["products", "shop", null]`로
각각 독립적으로 `useShopProducts(null)`를 호출한다.
React Query 캐시 관점에서는 하나로 합쳐지지만, 두 컴포넌트 모두
마운트 시점에 캐시가 비어 있으므로 **첫 요청은 둘 다 Firestore를 호출**한다.

**③ 렌더링-전략.md 결정과 실제 구현의 불일치**

문서에서 "상품 목록·메인 페이지 → SSR"로 명시했으나
실제 코드가 그 결정을 따르지 않고 있다.

## Task

- 서버에서 상품 데이터를 먼저 fetch하여 초기 HTML에 포함
- 클라이언트 hydrate 직후 데이터가 즉시 존재하도록 하여 skeleton 제거
- `MainHero`와 `ShopPage`의 중복 Firestore 요청 제거
- React Query 훅(`useShopProducts`) 인터페이스는 변경하지 않음

## Action

React Query의 공식 서버 prefetch 패턴인
**`prefetchInfiniteQuery` + `dehydrate` + `HydrationBoundary`** 를 사용한다.

### 흐름

```
page.tsx (서버 컴포넌트)
  → getQueryClient()로 서버 전용 QueryClient 생성
  → queryClient.prefetchInfiniteQuery(["products", "shop", null])
     → 서버에서 Firestore 직접 호출
  → dehydrate(queryClient) → JSON 직렬화
  → HydrationBoundary에 state로 전달

클라이언트 hydrate
  → MainHero / ShopPage가 useShopProducts 호출
  → React Query가 HydrationBoundary의 dehydrated state를 캐시로 복원
  → Firestore 추가 요청 없이 즉시 데이터 반환
```

### 핵심 변경

```tsx
// page.tsx
import { HydrationBoundary, dehydrate } from "@tanstack/react-query";
import { getQueryClient } from "@/lib/getQueryClient";
import { getShopProducts } from "@/services/product";

export default async function Home({ searchParams }) {
  const { category } = await searchParams;
  const queryClient = getQueryClient();

  await queryClient.prefetchInfiniteQuery({
    queryKey: ["products", "shop", null],
    queryFn: ({ pageParam }) => getShopProducts(null, pageParam),
    initialPageParam: undefined,
    getNextPageParam: (lastPage) => lastPage.lastDoc ?? undefined,
  });

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <MainHero />
      <ShopPage category={category ?? null} />
    </HydrationBoundary>
  );
}
```

### 서버 전용 QueryClient — `getQueryClient()`

`QueryProvider`의 `QueryClient`는 브라우저 메모리에서 인스턴스가 재사용되는 구조다.
서버 요청 간 상태 공유(요청 오염)를 막기 위해
서버 컴포넌트에서는 매 요청마다 새 인스턴스를 생성하는
별도 헬퍼(`src/lib/getQueryClient.ts`)를 사용한다.

React Query 공식 문서에서 권장하는 `cache()` 기반 싱글톤 패턴:

```ts
// src/lib/getQueryClient.ts
import { QueryClient } from "@tanstack/react-query";
import { cache } from "react";

export const getQueryClient = cache(
  () =>
    new QueryClient({
      defaultOptions: { queries: { staleTime: 60 * 1000 } },
    }),
);
```

`React.cache()`는 동일 서버 요청 내에서 인스턴스를 공유하고,
요청이 끝나면 자동으로 폐기하므로 요청 간 오염이 없다.

### `MainHero`와 `ShopPage` 내부 변경 없음

두 컴포넌트는 여전히 `useShopProducts`를 동일하게 호출한다.
변경되는 것은 **해당 queryKey의 캐시가 이미 채워져 있다는 사실**뿐이다.
`useInfiniteQuery`는 캐시에 데이터가 있으면 Firestore를 호출하지 않고
즉시 `{ status: "success", data: ... }`를 반환한다.

## Result

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| 초기 HTML | 상품 없음 (skeleton) | 서버에서 렌더링된 상품 포함 |
| SEO | 크롤러가 상품 읽지 못함 | 크롤러가 상품 데이터 읽음 |
| LCP | JS 실행 후 이미지 로드 | 서버 HTML에 이미지 src 포함 |
| 초기 Firestore 요청 | 클라이언트에서 2회 (MainHero + ShopPage) | 서버에서 1회 |
| 컴포넌트 인터페이스 | — | 변경 없음 |
| 무한 스크롤 | 클라이언트 동작 | 변경 없음 (유지) |

초기 렌더링 이후 동작(무한 스크롤, 카테고리 필터, 가격 정렬)은
모두 클라이언트에서 이전과 동일하게 동작한다.

> **참고**: Firestore 클라이언트 SDK는 브라우저 환경을 가정하므로
> 서버 컴포넌트에서 직접 호출할 경우 `firebase/app` 초기화가 서버 환경에서도
> 동작하는지 확인이 필요하다. 서버에서 Firestore Admin SDK를 사용하는 방식도
> 있으나 현재 프로젝트 범위에서는 클라이언트 SDK를 서버에서 그대로 호출하는
> 방향을 유지한다(`렌더링-전략.md` 참고).
