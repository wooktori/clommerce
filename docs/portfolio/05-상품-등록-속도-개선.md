# 상품 등록 속도 개선 — 추측이 아닌 실측으로 병목 찾기

## Situation

"상품 등록 속도가 느리다"는 리포트를 받았다. 코드만 보고 `createProduct`의 구조를
먼저 의심했다: 문서 생성(`addDoc`) → 이미지 업로드 → 문서 업데이트(`updateDoc`)가
전부 순차 대기하는 3단계 네트워크 왕복 구조였기 때문이다. "당연히 이게 느리겠지"라는
가정으로 바로 최적화 계획(쓰기 2번 → 1번 병합)을 세우고 사용자에게 승인까지 받았다.

그런데 문서화를 위해 개선 전 상태를 Chrome DevTools Network 탭으로 캡쳐해달라고
요청했고, 그 결과가 가설을 완전히 뒤집었다.

## Task

- 실측 데이터로 진짜 병목이 어디인지 다시 진단
- 진단 결과에 따라 최적화 방향을 재설정
- 이미지 업로드 전송량 자체를 줄이는 방법을 next/image 등 기존 도구로 해결할 수 있는지 검토

## Action

### 1. 실측 — Network 탭 Waterfall 분석

캡쳐된 요청을 `Initiator`(코드 라인) 기준으로 분류했다.

| 요청 | Initiator | 소요 시간 | 의미 |
|---|---|---|---|
| `channel?...` (Firestore 쓰기) | `product.ts:80`/`90` | 9~31ms | `addDoc`/`updateDoc` — **병목 아님** |
| `channel?gsessionid=...` | `product.ts:80` | 5.06s | Firestore WebChannel 최초 연결. **세션당 1회**만 발생, 등록 횟수와 무관 |
| `o?name=products%2F...` (Storage 업로드) | `product.ts:55` | **3.90s** | `uploadBytes` — 원본 1.6MB, 스로틀링 없음 |
| `products%2F...` (다운로드 URL 조회) | `product.ts:56` | 339ms | `getDownloadURL` |

가설이었던 Firestore 쓰기 2회는 합쳐도 40ms 수준밖에 안 되는 반면,
실제로 3.9초를 잡아먹는 건 이미지 업로드 그 자체였다.
**"구조가 비효율적으로 보인다"와 "실제로 느리다"는 다른 문제였다.**

### 2. next/image로 해결 가능한지 검토 후 배제

"이미지 최적화라면 next/image로 되지 않나?"라는 질문이 나왔지만, 동작 시점을 따져보면
불가능했다. next/image는 **이미 저장된 이미지를 요청 시점에 리사이즈해서 내려주는** 도구다.
반면 병목은 `<input type=file>`로 고른 원본을 **브라우저에서 Firebase Storage로 직접 업로드**하는
구간이고, 이 업로드는 Next.js 서버를 거치지 않는다. next/image가 개입할 지점 자체가 없었다.

→ 결론: 전송 바이트 자체를 줄이려면 **업로드 전에 브라우저에서 리사이즈**하는 수밖에 없다.

### 3. 대안 비교 — 압축 vs "선택 즉시 업로드"

용량을 줄이는 것 외에, 업로드 "시점"을 앞당겨 폼 입력 시간과 겹치게 하는 방법도 검토했다.
체감 속도 개선 폭은 더 크지만, 파일별 업로드 상태 관리·취소 처리 등 구현 복잡도가
이번 스코프에 비해 컸다. 사용자와 논의 후 이번엔 **리사이즈/압축만** 적용하고,
즉시 업로드는 별도 과제로 남기기로 했다.

### 4. 클라이언트 압축 구현 (`src/lib/image.ts`)

새 의존성 없이 `createImageBitmap` + `<canvas>`로 구현했다.
긴 변 1600px을 넘으면 축소하고 JPEG quality 0.8로 재인코딩한다.
이미 작은 이미지를 압축했을 때 오히려 커지는 경우를 대비해 결과를 원본과 비교해 더 큰 쪽은 버린다.

```ts
export async function compressImage(
  file: File,
  { maxDimension = 1600, quality = 0.8 } = {}
): Promise<File> {
  if (!file.type.startsWith("image/")) return file;

  const bitmap = await createImageBitmap(file);
  const scale = Math.min(1, maxDimension / Math.max(bitmap.width, bitmap.height));
  // ... canvas에 그린 뒤 toBlob으로 재인코딩
  if (!blob || blob.size >= file.size) return file; // 압축이 손해면 원본 사용
  return new File([blob], file.name.replace(/\.\w+$/, ".jpg"), { type: "image/jpeg" });
}
```

`uploadProductImages`(`src/services/product.ts`)에서 업로드 직전에 적용해
상품 등록·수정 양쪽에 자동으로 적용되게 했다.

### 5. addDoc + updateDoc → setDoc 병합 (부수 정리)

체감 속도 개선은 미미하다는 걸 이미 알고 있었지만, 업로드 중 "이미지 없는 문서"가
잠깐 노출되는 과도기 상태를 없애는 부수 효과가 있어 함께 적용했다.
`doc(collection(db, "products"))`로 네트워크 호출 없이 클라이언트에서 ID를 먼저 만들고,
업로드 완료 후 `setDoc` 한 번으로 최종 데이터를 쓰는 구조로 바꿨다.

### 6. 적용 후 재측정 — 압축이 예상만큼 안 먹힌 이유

배포 전에 실제 압축 바이트 수를 콘솔에 임시로 찍어 확인했다.

```ts
console.log(`[compressImage] ${file.name}: ${file.size}B -> ${compressed.size}B`);
// [compressImage] photo.jpg: 1,600,000B -> 157,428B
```

| 지표 | Before | After |
|---|---|---|
| 파일 크기 | 1.6MB | 154KB (**약 90% 감소**) |
| 업로드 소요 시간 | 3.90s | 3.39s (**약 13% 감소**) |
| Firestore 쓰기 요청 수 | `channel?...` 2세트(`:80`, `:90`) | 1세트(`:86`) — 병합 확인 |

바이트는 10배 넘게 줄었는데 시간은 13%밖에 줄지 않았다.
이건 이 3~4초짜리 지연이 **전송 대역폭이 아니라 고정 왕복 지연(latency)에 지배되고 있다**는 신호다.
압축(=페이로드 축소)은 대역폭이 병목일 때 효과적인 처방인데, 실제 병목은 다른 종류였던 것이다.

역설적으로 이 결과는 3단계에서 **복잡도 때문에 미룬 "선택 즉시 백그라운드 업로드"가
더 근본적인 해법이었다는 걸 재확인**시켜준다. 고정 지연은 압축으로 줄일 수 없지만,
업로드 시작 시점을 앞당겨 폼 입력 시간과 겹치면 사용자 입장에서는 그 지연 자체가 사라지기 때문이다.

## Result

- 진짜 병목(이미지 업로드)에 리소스를 집중해 원인과 무관한 최적화를 피함
- `compressImage`로 업로드 페이로드를 약 90% 줄였지만, 체감 속도 개선은 13% 수준에 그침 —
  병목이 대역폭이 아니라 고정 지연이었기 때문. 압축은 여전히 유효한 개선(느린 회선에서는 효과가 더 클 것)이지만
  이번 병목의 근본 해법은 아니었음을 재측정으로 확인
- `addDoc`+`updateDoc` 병합으로 과도기적 "이미지 없는 문서" 상태 제거 (Waterfall에서 쓰기 요청 2세트 → 1세트로 확인)
- **다음 과제**: "선택 즉시 백그라운드 업로드" — 고정 지연을 없애는 게 아니라 사용자 입력 시간 뒤로 숨기는 방식.
  이번 재측정으로 우선순위가 명확해졌으니 다음 스코프로 진행 예정
- **가장 큰 소득은 코드가 아니라 과정**: 코드만 보고 세운 최적화 계획을 실측으로 뒤집었고,
  적용 후에도 다시 실측해 "고쳤다고 믿는 것"과 "실제로 얼마나 나아졌는지"를 구분했다.
  최적화는 구현하고 끝나는 게 아니라 재측정으로 검증할 때 비로소 완결된다는 걸 확인한 작업이었다.
