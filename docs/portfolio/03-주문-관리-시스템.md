# 주문 관리 시스템 — 구매자/판매자 양방향 주문 흐름

## Situation

결제 완료 후 생성된 주문을 확인하거나 관리할 방법이 없었다.
구매자는 자신의 구매 내역을 볼 수 없었고, 판매자는 자기 상품의 주문이 들어왔는지조차 알 수 없는 상태였다.

또한 주문 상태를 나타내는 `OrderStatus` 타입이 문자열 리터럴(`"pending"`, `"delivered"`)로 분산 관리되어
컴포넌트마다 다른 값을 사용하는 타입 불일치 문제가 있었다.

## Task

- 구매자: 본인 주문 목록 조회, 주문 완료 상태에서만 취소 가능
- 판매자: 본인 상품에 대한 주문 목록 조회, 상태 단방향 변경
- `OrderStatus`를 중앙에서 일관되게 관리하는 타입 설계

## Action

### 1. OrderStatus 타입 설계 (`src/types/order.ts`)

`enum` 대신 `as const` 객체를 선택했다. `enum`은 트리쉐이킹이 되지 않아
번들에 불필요한 코드가 남는 반면, `as const` 객체는 타입과 값을 모두 추출할 수 있다.

```ts
export const ORDER_STATUS = {
  ORDER_COMPLETE:   "order_complete",
  PENDING_SHIPMENT: "pending_shipment",
  SHIPPING:         "shipping",
  CANCELLED:        "cancelled",
} as const;

export type OrderStatus = typeof ORDER_STATUS[keyof typeof ORDER_STATUS];
```

### 2. Firestore 복합 쿼리 + 인덱스

구매자/판매자 각각 `where` + `orderBy` 조합으로 쿼리한다.
Firestore에서 `where`와 `orderBy`를 함께 쓰면 복합 인덱스가 필수인데,
인덱스 없이 쿼리하면 에러가 발생한다.

Firebase 콘솔에서 두 개의 복합 인덱스를 수동 생성해 해결했다.

| 컬렉션 | 필드1 | 필드2 |
|---|---|---|
| `orders` | `buyerId` 오름차순 | `createdAt` 내림차순 |
| `orders` | `sellerId` 오름차순 | `createdAt` 내림차순 |

### 3. 구매자 주문 취소 — 재고 복구 통합

취소 시 주문 상태 변경과 재고 복구를 하나의 mutation에서 처리했다.
호출부가 두 작업을 각각 신경 쓰지 않아도 되도록 `useCancelOrder`에 캡슐화했다.

```ts
mutationFn: async ({ orderId, productId, quantity }) => {
  await cancelOrder(orderId);
  await restoreStock(productId, quantity);
},
```

취소 가능 조건은 `ORDER_COMPLETE` 상태일 때만으로 제한해
발송 시작 후 취소를 방지했다.

### 4. 판매자 주문 상태 단방향 변경

상태 변경은 `주문완료 → 발송대기 → 발송시작` 방향으로만 가능하다.
`NEXT_STATUS` 맵으로 다음 상태를 결정해 역방향 변경을 구조적으로 차단했다.

```ts
const NEXT_STATUS: Partial<Record<OrderStatus, OrderStatus>> = {
  [ORDER_STATUS.ORDER_COMPLETE]:   ORDER_STATUS.PENDING_SHIPMENT,
  [ORDER_STATUS.PENDING_SHIPMENT]: ORDER_STATUS.SHIPPING,
};
```

`SHIPPING` 이후에는 다음 상태가 없으므로 버튼 자체가 렌더링되지 않는다.

## Result

- 구매자: 본인 주문 목록 조회 및 주문완료 상태 취소 기능 완성
- 판매자: 본인 상품 주문만 필터링 조회, 단방향 상태 변경 완성
- `ORDER_STATUS` 상수로 타입 불일치 문제 해소 — 전체 코드베이스에서 일관된 값 사용
- Firestore 복합 인덱스 2개로 정렬된 쿼리 성능 확보
